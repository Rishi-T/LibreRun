# Filelist Generation & Optimization

LibreRun can automatically parse your RTL source directories, resolve module dependencies, and generate correctly-ordered filelists for Verilator — replacing manual filelist maintenance.

---

## Config Additions

The following fields are added to `rtl_configuration` in `base_config.yaml`:

```yaml
rtl_configuration:
  rtl_source_dirs:
    - $PROJECT_ROOT/rtl               # plain path; nickname = folder name ('rtl')
    - $PROJECT_ROOT/ip:core_ip        # nicknamed; output → autogen_core_ip.f
    - $PROJECT_ROOT/lib:core_ip       # same nickname → merged into autogen_core_ip.f

  file_exclude_patterns:              # glob patterns matched against filename
    - "*_stub*"
    - "*_tb*"

  folder_exclude_patterns:            # glob patterns matched against folder path components
    - "*sim*"
    - "*tb*"
```

- Paths support environment variable expansion
- Nicknames are optional — default is the folder name
- Multiple source dirs with the same nickname have their files merged into one `.f`
- `rtl_include_dirs` is **not used** — include directories are generated automatically inside filelists

---

## How `-flg` Works

### Step 1 — File Discovery
LibreRun recursively walks all `rtl_source_dirs` (unlimited depth), collecting `*.sv` and `*.v` files. Files and folders matching exclude patterns are skipped.

Each file is mapped to its source dir's nickname to determine which output `.f` it belongs to.

All folders and subfolders containing discovered files are collected for `-incdir` generation.

### Step 2 — Parallel Parsing
All discovered files are parsed in parallel using a thread pool. Before parsing, single-line (`//`) and block (`/* */`) comments are stripped to reduce false matches.

For each file, LibreRun extracts:
- **Module definitions** — via `MODULE_DEF_PATTERNS` regexes
- **Module instantiations** — via `MODULE_INST_PATTERNS` regexes, filtered against `SV_V_KEYWORDS`

This builds two registries:
- `module → [files defining it]`
- `file → [modules it instantiates]`

False positive dependencies are acceptable. Missing dependencies are not — the patterns err on the side of inclusion.

### Step 3 — Duplicate Detection
Any module defined in more than one file is a duplicate. LibreRun manages these via `misc/filelists/duplicate_mapping_<config_name>.yaml`.

If unresolved duplicates exist, filelist generation is aborted with an error. The full list of unresolved entries is written to the flow log; the terminal shows a count and the log path.

**See [Resolving Duplicates](#resolving-duplicates) below.**

### Step 4 — Dependency Resolution
Starting from `dut_top_module`, LibreRun performs a recursive DFS to build a topologically-sorted file list: dependencies appear before the files that depend on them.

Files never reached from `dut_top_module` (unreachable files) are placed at the **top** of the filelist as a safety measure, so they are compiled first and never silently omitted.

### Step 5 — Output
For each nickname group:
- A filelist is written to `misc/filelists/autogen_<nickname>.f`
- Format: `-incdir` lines first, then unreachable files, then dependency-ordered files

A marker file is written to `scripts/librerun_temp/.autogen_<config_name>` listing all generated filelist paths (relative to project root). Auto-generated filelists are included in compilation **only if this marker file exists**.

Manual filelists (`rtl_manual_filelists`) are always compiled first, unconditionally, regardless of whether `-flg` has been run.

## Filelist Format

```
# Auto-generated by LibreRun -flg. Do not edit manually.
-incdir /path/to/rtl/subdir_a
-incdir /path/to/rtl/subdir_b

# Files not reached from dut_top (placed here for safety)
/path/to/rtl/possibly_unused.sv

/path/to/rtl/pkg_definitions.sv
/path/to/rtl/submodule_a.sv
/path/to/rtl/submodule_b.sv
/path/to/rtl/top.sv
```

## Filelist Optimization (`-flo`)

`-flo` must be used together with `-flg`. It prunes unreachable files from the generated filelists:

- Files not reachable from `dut_top_module` are removed entirely (not placed at the top)
- `-incdir` entries for folders that have no remaining files after pruning are also removed
- Terminal output: count of pruned files and directories
- Full pruned file list: written to the flow log only

```bash
lr -flg -flo
```

> Use with caution — if the dependency regex misses any instantiation, pruning may remove a file that is actually needed. Run without `-flo` first, verify the design compiles, then prune.

## Resolving Duplicates

When multiple files define the same module name, LibreRun creates or updates `misc/filelists/duplicate_mapping_<config_name>.yaml`:

```yaml
my_module:
  file_to_use: "Not Set"      # set to 0-based index of the file to use
  files:
    - rtl/subdir_a/my_module.sv
    - rtl/subdir_b/my_module.sv
```

Edit `file_to_use` to the index (0, 1, 2...) of the file you want to use, then re-run `-flg`.

### Mapping Preservation Between Runs

LibreRun preserves your choices intelligently between `-flg` invocations:

| Situation | Behaviour |
|-----------|-----------|
| Module newly becomes a duplicate | New entry added, `file_to_use: Not Set` |
| Module no longer a duplicate | Entry removed |
| Same files, mapping already set | Mapping preserved |
| Same files, mapping not set | Preserved as `Not Set` |
| File removed, chosen file unaffected | Index adjusted to new position |
| File removed, chosen file was the one removed | Reset to `Not Set` |
| New file added to an already-mapped duplicate | Reset to `Not Set` (safest) |

Changes to the mapping between runs are written in detail to the flow log. The terminal shows a count and the log path.

## Extending Detection

The regex patterns and keyword list that drive parsing are defined as named constants at the top of the filelist generation section in `librerun.py`, and are designed to be easy to extend:

```python
# Add new module definition patterns here
# Each pattern must yield the module name in group(1)
MODULE_DEF_PATTERNS = [
    re.compile(r'^\s*module\s+(\w+)', re.MULTILINE),
]

# Add new instantiation detection patterns here
# Each pattern must yield the instantiated module name in group(1)
MODULE_INST_PATTERNS = [
    re.compile(r'^\s*(\w+)\s+(?:#\s*\(|(\w+)\s*\()', re.MULTILINE),
]

# Add SV/V reserved words here to suppress false dependency detection
SV_V_KEYWORDS = frozenset([
    "module", "always_ff", "logic", ...
])
```

To improve detection for a new construct, add a pattern to the appropriate list. No other changes are needed.
